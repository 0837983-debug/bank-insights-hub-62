### D.9 — Оптимизация загрузки в STG (batch insert) ✅ ЗАВЕРШЕНО
**Смысл:** ускорить загрузку данных в STG с 40 сек до ~300мс для 500 строк.
- ✅ Реализован batch insert через `unnest` с батчами по 1000 строк
- ✅ `loadToSTG()` — Balance
- ✅ `loadFinResultsToSTG()` — Financial Results
- ✅ Результат: 500 строк загружаются менее чем за 1 секунду

- **Проблема (D.8):** при загрузке файла с периодами, которые уже есть в ODS/MART, данные молча перезаписываются.
- **Решение:**
  1. После парсинга файла — извлечь уникальные периоды из данных.
  2. Проверить в ODS/MART наличие данных за эти периоды.
  3. Если данные существуют — вернуть предупреждение на фронт.
  4. UI: показать диалог "Данные за период(ы) X уже существуют. Перезаписать?"
  5. Пользователь подтверждает или отменяет.
  6. При подтверждении — пометить все записи за эти периоды как удалённые (soft-delete) в ODS и MART.
  7. Загрузить новые данные по стандартной процедуре.
- **Backend:**
  - Новый endpoint: `POST /api/upload/check-periods` — проверка периодов.
  - Параметр `force: true` в upload для принудительной перезаписи.
  - Функция `softDeleteByPeriods(periods[], tableName)`.
- **Frontend:**
  - Компонент `OverwriteConfirmDialog`.
  - Логика: upload → check-periods → (если есть конфликты) → диалог → upload с force.
- **Применимо к:** Balance, Financial Results, любые будущие источники.
- Проверка: при загрузке данных за существующий период появляется диалог подтверждения.
